# Imagine Deck Web Redesign Implementation Plan

> 対象サイト: https://imaginedeck.igsrr.org/
> 参考デザイン: https://yoshidakikaku.jp/
> 作成日: 2026-05-10
> 想定読者: 現場エンジニア / Imagine Deck 運営

---

## 1. 改築の目的

Imagine Deck は、愛媛大学ミュージアムに新設された **「知的交流のハブ」** として機能する交流スペースである。
愛媛大学ミュージアムの 3 つの基本機能（研究成果の発信／社会教育への地域参加／学生の学習支援）をさらに発展させるべく、Imagine Deck は **4 つのコア機能**（オープンスペース／ショーケース／ワークショップ＆スタートアップ支援／インタラクティブ体験）を担い、「見る・触れる・考える・対話する」という 4 つの新しい柱を中心に設計されている。

現在の公式サイトは、予約フォーム・ガイドライン・カレンダー・開催ログが機能単位で分かれており、訪問者は「Imagine Deck がどんな場所で、自分が何をできるのか」を読み解く前にフォームに到達してしまう。これは、本来このスペースが持つ「ミュージアム × 大学 × 地域コミュニティ」の文脈・温度を伝えきれていない。

今回の改築のゴールは次の 3 点である。

1. **場の魅力を最初の 1 画面で伝える** — 訪れた瞬間に「ここで何ができるのか」「自分が来てよい場所か」が判る。
2. **行動への自然な導線を設計する** — 予約、開催ログ閲覧、ガイドライン参照、問い合わせの 4 動線を、ストーリー順に並べて自然に進ませる。
3. **運営の活動アーカイブを資産化する** — 開催ログを単なる投稿一覧ではなく「Imagine Deck が地域・大学に与えた価値」を可視化するアーカイブにする。

実装スタックは **HTML / CSS / Vanilla JavaScript** のみ。フレームワーク（React, Next.js 等）には依存しない。
理由: 既存サイトへの段階的組み込み、運営側の保守容易性、静的ホスティングの軽量性、将来別技術への移植容易性のため。

## 2. 現行サイトの課題整理

ヒアリングおよび `https://imaginedeck.igsrr.org/` のページ構成（ホーム / 予約 / ガイドライン / カレンダー / 開催ログ）から導出した課題。

### 2.1 トップページ

- **「何の場所か」が冒頭で伝わらない。** ロゴと名称のみが先行し、価値提案（誰のための、何ができる場所か）が言語化されていない。
- **トップから他ページへの導線がフラットで、優先順位が無い。** 「予約」「開催ログ」が同列扱いで、初訪者が「まず読むべきもの」が見えない。
- **ビジュアルが弱い。** ポスターで使われている紫・猫・「見る／触れる／考える／対話する」の世界観が Web に転写されていない。

### 2.2 予約フォーム

- 必須項目が **1 画面に縦長で並び**、初見の学生にとって心理的負担が大きい。
- 各入力欄に **入力例・説明文が無い** ため、「学術イベントかどうか」「利用箇所」「予約形態」など Imagine Deck 特有の概念が判らない。
- **送信前の確認画面が無い／弱い**。誤送信時のリカバリが効かない。
- **エラーは送信後に表示されがち**で、ステップごとの即時フィードバックになっていない。
- **利用箇所が単純なチェックボックス羅列**。本来は空間配置（壁面／フロア／カウンター等）に紐づくため、視覚的なマッピングと相性が良いはず。

### 2.3 ガイドライン

- **本文が長く、優先度が読み取れない。** 「まず守ること」と「詳細ルール」が同じ濃度で並ぶため、初見者がどこから読めば良いか判らない。
- **検索・アンカー導線が弱い**。特定ルールに直接ジャンプできない。

### 2.4 カレンダー

- 月表示はあるが、**予約済み日の色分け／イベント種別の色分けが無い／弱い**。
- 「**この日で予約する**」のような行動誘導が無い。閲覧と予約が断絶している。

### 2.5 開催ログ

- 投稿一覧として並ぶのみ。**カードの情報設計が薄い**（種別・参加人数・成果といった "実績指標" が前面に出ていない）。
- **タグフィルター・年月フィルター・検索が無い／弱い**。アーカイブが増えると埋もれる。
- 詳細ページの **ストーリーテリング構造が無い**（概要 → 当日の様子 → 反応 → 学び の流れになっていない）。

### 2.6 全体

- ブランドの一貫性（紫・猫・「ただいま準備中」の手描き温度感）が **トップ以外で消える**。
- スマホファーストで設計されておらず、特に予約フォームは **375px 幅で崩れやすい**。

## 3. 参考デザイン分析

### 3.1 吉田企画サイト（https://yoshidakikaku.jp/）から取り入れる要素

吉田企画は「PRECIOUS DESIGN!」という大見出しで世界観を提示し、以下の構造で訪問者を導いていた。

```
Hero（PRECIOUS DESIGN! + 事業内容ワンライナー）
↓ Message（"今、必要なデザインを『一緒に』作ります"）
↓ Service（Design / Construction / Consultation の 3 段）
↓ 4,000店舗以上 という実績数値で信頼を可視化
↓ Works（業種別事例を画像で並べる）
↓ Contact
```

**取り入れる点:**

| 吉田企画の特徴 | Imagine Deck への翻訳 |
| --- | --- |
| ファーストビューで世界観を 1 行で提示 | 「Imagine the Future. 見る、触れる、考える、対話する。大学の知を開く交流スペース。」を Hero に大きく置く |
| 「一緒に作る」という共創姿勢の言語化 | 「学生・研究室・地域が一緒に知を開く場所」という温度感に置換 |
| 数値で実績を可視化（4,000店舗以上） | 「累計開催イベント数 / 主催学生数 / 関わった研究室数」を運営が更新できる数値カードに |
| Works セクションで業種別に事例を画像で並べる | 開催ログのカードを、ジャンル別画像グリッドとして見せる |
| セクション境界の明瞭さ・余白の広さ | セクション間に最低 96px の余白、見出しは hero 64px / section 40px |
| 「上へ戻る」ボタン | 長スクロールでも迷子にならないよう常設 |

**取り入れない点（Imagine Deck の文脈に合わない部分）:**

- 派手な看板・施工写真の多用 → Imagine Deck はミュージアム文脈なので、静謐さ・余白・大学の知性を優先。
- 大文字英語見出しの強さ → 「Imagine the Future.」程度に留め、日本語見出しを主役にする。

### 3.2 ChatGPT 風 UI から取り入れる要素

ChatGPT のフロントエンド思想は「**情報の段階的開示**」「**広い余白**」「**会話的な進行**」にある。

| ChatGPT の思想 | Imagine Deck への適用 |
| --- | --- |
| 余白を広く取り、必要な情報だけをカード単位で提示 | 予約フォームを縦長 1 画面にせず、ステップごとにカード分割 |
| 会話的・段階的な入力フロー | 予約フォームを 6 ステップに分割（後述） |
| プレースホルダーや補助テキストで操作を誘導 | 各入力欄に "例：研究成果の展示、学生向けワークショップ..." を併記 |
| 中央寄せの読みやすい本文幅 | 本文最大 720px、カード最大 960px、画面最大 1280px |
| 暗黙的なエラー回避（入力中検証） | ステップ進行ボタン押下時に当該ステップ内をバリデート、進めない場合は箇条書きでエラー表示 |
| 落ち着いたモノクローム + アクセント色 1 つ | ベージュ背景 + 紫アクセント。色は「行動・状態」のために使い、装飾には使わない |

## 4. 新サイトの情報設計

### 4.1 ページ構成

```
/                  index.html         トップ（全体像が掴める LP）
/reserve           reserve.html       予約（6 ステップフォーム）
/guidelines        guidelines.html    ガイドライン（要約 → 詳細）
/calendar          calendar.html      カレンダー（月表示）
/event-log         event-log.html     開催ログ一覧
/event/:id         event-detail.html  開催ログ詳細（テンプレート）
```

### 4.2 導線設計

トップページのストーリー順:

```
Hero（場の世界観）
 ↓ Imagine Deck とは（誰の何の場所か）
 ↓ ここでできること（4 機能カード）
 ↓ 利用の流れ（予約 → 当日 → 振り返り）
 ↓ 実績数値（運営更新可能）
 ↓ 最新の開催ログ（3 件）
 ↓ 予約 CTA（大）/ ガイドライン CTA（小）
 ↓ お問い合わせ
```

主要 CTA は 2 つに絞る:

1. **スペースを予約する** → `/reserve`（主動線）
2. **開催ログを見る** → `/event-log`（信頼形成動線）

サブ動線として、Header にナビ、Footer にサイトマップを置く。

### 4.3 ユーザー種別と各々の優先動線

| ユーザー | 主目的 | 優先動線 |
| --- | --- | --- |
| **学生**（個人） | 自習・友人との対話 | トップ → ここでできること → ガイドライン（予約不要の確認） |
| **学生**（主催者） | イベント・展示開催 | トップ → 利用の流れ → 予約 |
| **教職員** | 授業外活動・研究室企画 | トップ → 開催ログ（前例参照） → 予約 |
| **研究室** | 成果展示・ワークショップ | 開催ログ → 詳細 → 予約 |
| **地域関係者** | 地域連携イベント | トップ → Imagine Deck とは → 問い合わせ（学外者は有料案内） |
| **運営者** | 開催ログ更新・状況把握 | カレンダー → 開催ログ更新（将来的に CMS 化） |

## 5. ページ別仕様

### 5.1 トップページ（index.html）

#### セクション構成

1. **Hero**
   - 大見出し: 「Imagine the Future.」
   - サブ見出し: 「見る、触れる、考える、対話する。大学の知を開く交流スペース。」
   - リード文: 「Imagine Deck は、愛媛大学ミュージアムに生まれた、学生・研究室・地域が交わる新しい居場所です。」
   - CTA: ［スペースを予約する］（primary）／［開催ログを見る］（ghost）
   - ビジュアル: 紫グラデーション背景 + 雪・猫モチーフのライン装飾 + 「ただいま準備中」のスタンプ要素（ポスター由来）

2. **Imagine Deck とは**
   - 1 枚の本文カード。「ミュージアムの一角に生まれた、開かれた知の交流空間」という説明 + 場所写真。

3. **4 つのコア機能（公式版）**
   - ① **オープンスペース機能** — 多様な人々が自習・対話・交流できるコモンズ
   - ② **ショーケース機能** — 研究科・学部・センターの研究成果をプロトタイプ／映像で展示し、市民フィードバックを得る窓口
   - ③ **ワークショップ ＆ スタートアップ支援機能** — 学生主催 WS、PBL、起業支援
   - ④ **インタラクティブ体験** — 「見る・触れる・考える・対話する」多面的な体験
   - 各カードに英語ラベル（Open Space / Showcase / Workshop & Startup / Interactive）を併記。

4. **利用の流れ（3 ステップ）**
   - ①予約する／②当日利用する／③開催ログに記録する
   - 各ステップにアイコン + 1 行説明 + 該当ページへの小 CTA。

5. **数字で見る Imagine Deck**
   - 「累計開催イベント ◯件 / 主催学生 ◯人 / 関わった研究室 ◯室」
   - 数値は HTML に直書きしつつ、運営が更新しやすいよう `<span data-stat="events">` 形式でセマンティクスを付ける。

6. **最新の開催ログ（3 件）**
   - カードグリッド。`event-log.js` で生成、または静的に直書き（初期実装は静的）。
   - 「すべて見る」リンクで `/event-log` へ。

7. **予約 CTA バナー**
   - 紫背景 + 大ボタン「スペースを予約する」+ 補足「学生は無料・学外主催者は有料」。

8. **ガイドラインへの導線**
   - 「初めての方へ。利用ルールを 1 分でチェック。」 + 小 CTA。

9. **お問い合わせ**
   - メール + 申請フォームへの導線（運営問い合わせ用）。

#### 必須要素

- ヘッダー（透過 → スクロールで白背景）
- フッター（サイトマップ + クレジット + ミュージアム公式リンク）
- スマホ時はハンバーガーメニュー

### 5.2 予約ページ（reserve.html）

#### 5.2.1 予約フォームの人間工学的改善方針

予約フォームでは、現行フォームの情報を**削らず**、入力順序・表示形式・説明文を改善する。
利用者が最初から全項目を見て圧迫感を覚えないよう、6 ステップ構成を採用する。原則は次の 4 つ。

- **Structure（構造化）**: 項目を意味で分け、各ステップは 1 つの目的だけを担う。
- **Transparency（透明性）**: なぜこの情報が必要かを各ブロック冒頭で説明する。
- **Clarity（明瞭性）**: 専門用語を避け、必ず例示を添える。
- **Support（支援）**: 入力中・送信前にエラーを取り除き、確認画面で最終チェックする。

#### 5.2.2 予約フォームの情報分類（情報を削らず再編する）

5 つの情報カテゴリに分類し、ステップに割り当てる。

##### A. 予約可能性を判断する情報 → Step 1
- 日付
- 開始時間
- 終了時間

##### B. 責任者を特定する情報 → Step 2
- 主催者・団体名
- 担当学生氏名
- 担当学生メールアドレス
- 担当教員・担当職員名
- 担当教員・担当職員メールアドレス

##### C. 利用内容を判断する情報 → Step 3
- 予約形態（通常／専有／調整中）
- 参加形態（一般公開／学内のみ／関係者のみ）
- 学術イベントかどうか
- 想定人数
- 参加費の有無
- 飲食の有無

##### D. 空間利用を判断する情報 → Step 4
- 壁面（入口側）
- 壁面（奥）
- フロア（入口側）
- フロア（奥）
- カウンター
- その他

##### E. 運営確認に必要な詳細情報 → Step 5
- 利用目的（自由記述）
- イベント詳細（自由記述）
- SNS リンク
- 関連 URL
- 添付資料（複数）
- 備考

#### 5.2.3 6 ステップ構成

各ステップは 1 カードに収める。Stepper を上部に常設し、現在ステップ・完了ステップ・エラーステップを視覚化する。

```
Step 1  予約できるか確認する     [日付選択カレンダー / 開始時間 / 終了時間 / 空き状況フィードバック]
Step 2  主催者情報を入力する     [団体名 / 担当学生氏名・メール / 担当教員氏名・メール]
Step 3  利用内容を選ぶ           [予約形態 / 参加形態 / 学術 / 人数 / 参加費 / 飲食]
Step 4  利用箇所を選ぶ           [カード型チェックボックス（将来見取り図化）]
Step 5  詳細・添付資料を入力する [利用目的 / イベント詳細 / SNS / 関連URL / 添付 / 備考]
Step 6  確認して送信             [サマリ表示 / 各セクション「修正」 / 送信ボタン]
```

##### Step 1：まず「予約できるか」を確認する

利用者が最初に知りたいのは「自分の希望日時にここを使えるのか」。**入力項目より先に、可否判定を出す。**

- カレンダー UI（calendar.js を再利用）。予約済み日は色分け、休館日は disable。
- 開始時間・終了時間は 30 分刻みプルダウン（10:00〜16:30）。営業時間外・終了 ≦ 開始は選べない／即警告。
- 既存予約と重複する場合はステップ内で警告（「14:30〜15:30 と重複しています」）。
- 空きが確認できると次のフィードバックを表示してから「次へ進む」を活性化する。

> 5月18日（月）14:00〜17:00 は予約可能です。このまま予約内容の入力に進めます。

**人間工学的意図**: 無駄な入力を防ぐ／早い段階で不安を取り除く。

##### Step 2：主催者情報

「誰が責任を持つのか」を明確にする。学生にとって「担当教員」「団体名」は迷いやすいので、必ず説明文を付ける。

- 入力欄は 1 カラム、必須項目には `[必須]` ラベル。
- メールは `type="email"` + 即時形式チェック（blur 時）。
- 各項目の冒頭に「なぜ必要か」を 1 行で書く。

例（補足文）:

> 担当教員・担当職員は、企画内容を確認できる大学関係者を記入してください。
> 研究室・授業・ゼミ・学生団体で利用する場合は、代表者と確認者を分けて記入してください。

##### Step 3：利用内容

文字入力より先に、**選択式で大枠を決める**。判断負担を下げ、運営側の分類も容易にする。

- 「予約形態」「参加形態」「学術／非学術」「飲食有無」「参加費有無」は **カード型ラジオ**。
- 想定人数は数値入力（1〜100、Imagine Deck の収容を超える場合は警告）。
- カードは 1 枚あたり 88px 以上の高さで、見出し + 説明文を入れる。

例（カード）:
```
[展示・発表]    研究成果、制作物、活動紹介などを展示する
[イベント・交流] トークイベント、説明会、交流会などを行う
[ワークショップ] 参加者が手を動かして学ぶ企画
[自習・打ち合わせ] 少人数での学習・相談・打ち合わせ
```

##### Step 4：利用箇所

**カード型チェックボックス**（複数選択可）。色だけでなく ✓ アイコンで選択を伝える。
将来 SVG 見取り図 UI に差し替えるため、HTML は次の構造で書く（後述 components.css `.c-choice-card`）。

```html
<label class="c-choice-card" data-area="wall-front">
  <input type="checkbox" name="area" value="wall-front" hidden>
  <span class="c-choice-card__visual"><!-- 将来 SVG --></span>
  <span class="c-choice-card__title">壁面（入口側）</span>
  <span class="c-choice-card__desc">ポスター展示、来場者への案内、入口付近の掲示に向いています。</span>
</label>
```

各カードに具体的な利用例を添える:

- **壁面（入口側）**: ポスター展示、来場者への案内、入口付近の掲示に向いています。
- **壁面（奥）**: 落ち着いた展示、長時間の掲示に向いています。
- **フロア（入口側）**: ライトな立ち話・受付・物販に向いています。
- **フロア（奥）**: ワークショップ、少人数イベント、机を使う活動に向いています。
- **カウンター**: 受付・ドリンク提供・物販に向いています。
- **その他**: 上記以外の利用形態の場合（備考欄に詳細を）。

##### Step 5：詳細・添付資料

入力負荷が最も高いステップ。**例文を先に見せる**。

- `<textarea>` には `placeholder` ではなく **欄外のヘルプテキスト**で例文を出す（プレースホルダは入力中に消えて読めなくなるため）。
- ファイル添付 UI は drag&drop **と** クリック選択の両方をサポート（WCAG 2.2 ドラッグ操作非依存）。
- ファイルタイプ・最大容量を明示（PDF / PNG / JPG / 各 10MB まで）。

入力補助例:

> **利用目的**: 例：研究成果の展示、学生向けワークショップ、地域交流イベント、授業外活動の発表など
>
> **イベント詳細**: 当日の流れ、展示物、参加対象者、必要な備品、運営体制などを記入してください。
>
> **添付資料**: 企画書、ポスター案、展示レイアウト案、タイムテーブルなどがあれば添付してください。

##### Step 6：確認して送信

送信前確認は **必須**。ユーザーが最後に「この内容で良い」と判断できる画面にする。

- 5 セクション（日時／主催者／利用内容／利用箇所／詳細）をカードで表示。
- 各カードに `[修正]` ボタンを置き、押下で該当ステップに戻る（入力内容は保持）。
- 送信ボタンは状態を 4 段階で表す:
  - 通常: `予約リクエストを送信する`
  - 送信中: `送信しています...`（disabled, スピナー）
  - 成功: `送信が完了しました`（成功カードを表示）
  - 失敗: `送信に失敗しました。時間をおいて再度お試しください`（再送信導線）

#### 5.2.4 入力補助文の設計ガイドライン

各項目に必ず以下を併記する:

1. **なぜ必要か**（1 行）
2. **入力例**（1〜2 例）
3. **書き方のヒント**（必要な場合のみ）

専門用語は使わず、初見の学生でも理解できる言葉に翻訳する。

#### 5.2.5 バリデーションルール（reservation.js 参照）

- 「次へ」押下時に **当該ステップのみ** バリデート（先に進ませない）。

## 9. JavaScript 設計方針

### 9.1 全体方針

- **Vanilla JS**、ES Modules（`<script type="module">`）。
- 1 機能 1 ファイル、グローバル汚染なし。
- `DOMContentLoaded` で初期化、初期化関数は冪等に。
- `data-*` 属性で振る舞いを宣言（HTML を読めば挙動が判る）。

### 9.2 各モジュールの責務

#### `main.js`

- ハンバーガーメニュー（`[data-menu-toggle]` で開閉）
- ページ内アンカースムーススクロール
- 現在ナビの `aria-current="page"` 設定
- ヘッダーのスクロール変色（IntersectionObserver）
- 言語切替の **土台のみ**（`<html lang>` と簡単な辞書置換、初期実装は ja のみ）

#### `reservation.js`

##### 状態管理（モジュール内オブジェクト）

```js
const state = {
  currentStep: 1,            // 1〜6
  formData: {},              // 全フィールドの値
  validationErrors: {},      // step 番号 → エラー配列
  uploadedFiles: [],         // {name, size, type, blob}
};
```

##### ステップ移動

- 「次へ」押下時に **現在ステップのみ** 検証。エラーがあれば移動させず、エラーサマリにフォーカスする。
- 「戻る」押下時は **入力内容を保持**したまま 1 つ前のステップへ。
- Step 6 から各セクションの「修正」を押すと該当ステップにジャンプ（formData は保持）。

##### バリデーション（共通ルール）

| ルール | 適用例 |
| --- | --- |
| `required` | 主催者氏名、各メール、利用箇所（最低 1 つ）、利用目的 |
| `email` | 担当学生・担当教員のメール |
| `timeRange` | 終了時間 > 開始時間 + 営業時間内（10:00〜16:30） |
| `maxLength` | textarea は 2000 字、自由記述は 200 字 |
| `fileType` | PDF / PNG / JPG / GIF |
| `fileSize` | 1 ファイル 10MB 以下、合計 30MB 以下 |
| `numberRange` | 想定人数 1〜100 |

##### エラー表示

- 入力欄直下に赤色テキスト + アイコン（色だけに頼らない）。
- ステップ上部に **エラーサマリ**（リンク式：クリックで該当欄へフォーカス）。
- `aria-invalid="true"` を該当 input に付与、`aria-describedby` でメッセージと紐付け。
- ステップが切り替わったときにエラーサマリへ `tabindex="-1"` でフォーカスを当てて読み上げる。

##### 確認画面生成（Step 6）

- `formData` から確認用 HTML を生成（5 セクション）。
- 各セクションに「修正」ボタン → `state.currentStep` を変更して該当ステップへ。
- 値は HTML エスケープして表示（XSS 対策）。

##### 送信処理

- submit 中はボタンを `disabled` + `aria-busy="true"`。
- 送信先 URL は `<form data-endpoint="...">` で差し替え可能。
- 成功時: 完了画面（カード）+ 「開催ログを見る」「ガイドラインを見る」CTA。
- 失敗時: 失敗カード + 「再送信する」「内容を確認する」CTA。
- ネットワークエラー時はローカル下書きを `localStorage` に退避（次回起動時に復元提案）。

##### URL クエリ受け取り

- `?date=YYYY-MM-DD` を受け取り Step 1 のカレンダーを自動選択。
- `?start=HH:MM&end=HH:MM` も受け取り、時刻プルダウンを自動設定。

#### `calendar.js`

- 純粋な月カレンダー描画（`Date` のみ）
- 予約データはローカル定数 or `/data/events.json`（fetch）
- 日付クリック → モーダル
- 「この日で予約する」 → `/reserve?date=...` へ遷移

#### `event-log.js`

- カード DOM を読み取り、`data-*` 属性でフィルタリング（DOM 直操作、再描画なし）
- 検索: タイトル + 概要に対する部分一致（小文字化）
- タグフィルター: 複数選択 OR
- 年月フィルター: ラジオ
- 全フィルター AND で適用
- フィルタ結果が 0 件の時は空状態 UI を表示

### 10.2 各ページの注意点

- **予約フォーム**: モバイルでは Stepper を縦置き（左にステップアイコン、右に内容）。

- ボタン・リンクの最小タップ領域: 44 × 44px（iOS HIG 準拠）。

## 11. アクセシビリティ対応方針

- すべての画像に `alt` を付ける（装飾は `alt=""` + `role="presentation"`）。
- フォームの `<label>` を全項目に。視覚的に隠したいなら `.u-sr-only` で読み上げのみ残す。
- フォーカスリングを消さない（`:focus-visible` でブランド色のリング）。
- 色コントラスト比 4.5:1 以上（特に `--color-text-muted` の on `--color-bg` を確認）。
- ステッパーは `aria-current="step"` で現在ステップを伝える。
- モーダルは `role="dialog"` + `aria-modal="true"`、開いた時にフォーカスをモーダル内へ移動、Esc で閉じる、フォーカストラップを実装。
- ハンバーガーメニューは `aria-expanded` を更新。
- カレンダーのセルは `<button type="button">` にして、上下左右キー移動を `keydown` で実装（次フェーズ）。
- 状態（成功 / エラー）はアイコン + テキストで伝える（色だけに頼らない）。
- 動画・自動再生コンテンツは置かない（眩しさ・聴覚配慮）。

## 16. 人間工学・UX 設計原則（全ページ共通）

本プランの中核思想。トップから予約完了までの全画面で、以下を守る。

### 16.1 認知負荷を下げる設計（NN/g の Form Usability 原則を Imagine Deck 文脈に翻訳）

- **Structure（構造化）**
  - フォームは「文章を読む作業」ではなく「判断と入力を繰り返す作業」。
  - 項目を意味で分け、各ステップ／カードは 1 つの目的だけを担う。
  - 関連する項目は隣接させ、視線移動を最小化する。
- **Transparency（透明性）**
  - 「なぜこの情報が必要か」を各ブロック冒頭で説明する。
  - 例: 「担当教員のメールは、企画内容の確認のために運営が連絡することがあります。」
- **Clarity（明瞭性）**
  - 専門用語を避け、必ず例示を添える。
  - ラベルは名詞だけで終わらせず、誰が何を書く欄か一見してわかる文にする。
- **Support（支援）**
  - 入力中（blur）と送信前（確認画面）の 2 段階で支援する。
  - エラーは発見した瞬間に表示し、修正方法も合わせて提示する。

### 16.2 段階的開示（Progressive Disclosure）

複雑な UI で必要な情報を必要なタイミングで見せ、認知負荷を下げる手法。**項目を削るのではなく、表示順序と表示方法を整理する**。

- **最初に見せる**: 予約日 → 主催者情報 → 利用目的の大枠
- **入力が進んだら見せる**: 詳細情報 → 添付資料 → 備考 → 確認画面
- **特定条件でだけ見せる**: 「学術イベント」を選んだら学術関連の補足、「飲食あり」なら飲食ルール、など
- 段階的開示しても **運営上必要な情報は削らない**。順番・表示方法・説明文だけを改善する。

### 16.3 エラー予防（Error Prevention > Error Message）

可能な限り、エラーが起きない設計をする。エラーが避けられない時は、その場で支援する。

- 送信後ではなく **入力中** に支援する。
- 終了時刻 ≦ 開始時刻、営業時間外などは **選べない／即時警告**。
- 必須項目の未入力は、次のステップに進もうとした瞬間にその場で表示。
- 送信前確認画面を **必須**。送信ボタンを押す前に全内容を見直せる。
- 送信中の二重クリックを防ぐ（送信ボタンを `disabled` に）。
- ネットワーク失敗時は `localStorage` に下書き退避。

### 16.4 スマホ操作（モバイルファースト）

- タップ領域は **44 × 44px 以上**（iOS HIG / WCAG 2.2 Target Size 準拠）。
- フォームは **1 カラム**（横並びにしない）。
- ソフトキーボード表示時に入力欄が隠れないよう、`scrollIntoView({block: 'center'})` で当該欄を中央へ。
- ボタンは下部固定も検討する（ステップ進行ボタンは長いフォームで特に有効）。
- フォントサイズは 16px 以上（iOS でズームを発生させないため）。

### 16.5 アクセシビリティ（WCAG 2.2 AA を目標）

- 色だけで状態を伝えない（必ずアイコン + テキストを併用）。
- キーボード操作で全機能が使えること。
- フォーカス表示を消さない（`:focus-visible` でブランド色のリング）。
- ドラッグ操作に依存しない（ファイル添付はクリック選択にも対応）。
- フォーム項目すべてに `<label>` を付ける。
- ステップは `aria-current="step"`、現在ステップにフォーカスを当てる。
- モーダルは `role="dialog"` + `aria-modal` + フォーカストラップ + Esc で閉じる。
- 動きを抑える: `prefers-reduced-motion` を尊重し、アニメーションを停止可能にする。

### 16.6 信頼形成（Trust Building）

- 数値で実績を可視化（累計開催数 / 主催学生数 / 関わった研究室数）。
- 開催ログのアーカイブを充実させ、「過去にここで何があったか」を読ませる。
- 写真・引用で当事者の声を伝える。
- 運営の連絡先・問い合わせ先を分かりやすい位置に。

### 16.7 ブランド一貫性（吉田企画 × ChatGPT 風 UI の統合）

- **吉田企画から取り入れる**: 大胆な見出し、世界観の言語化、実績の数値・画像での可視化、セクション境界の明瞭さ。
- **ChatGPT 風から取り入れる**: 広い余白、カード単位の段階的表示、会話的進行、落ち着いたコントラスト、補助テキストの丁寧さ。
- 両者の統合点 = **「ミュージアムの静謐さと、学生の入りやすさを同居させる」**。

### 16.8 重要原則のサマリ（実装中の判断基準）

迷ったら以下を思い出す。

1. 予約フォームの項目は **削らない**
2. ただし、表示順序と見せ方を改善する
3. 1 画面に全項目を出さない
4. ステップごとに「今やること」を明確にする
5. 入力例を増やす
6. エラーをその場で出す
7. 送信前確認を必須にする
8. スマホで使いやすくする
9. 運営側が確認しやすい情報構造にする
10. 吉田企画のデザイン性と ChatGPT 風の操作性を統合する
